# Procurement Knowledge Base — MCP Server

A local data pipeline and MCP server that allows AI agents to retrieve, reason over, and explain information from a procurement document knowledge base.

Built as a take-home exercise for Caseware. Designed to be simple, local, and maintainable — not overengineered.

---

## What it does

Ingests 45 procurement documents (invoices, purchase orders, shipping orders, inventory reports, and one contract), extracts structured fields and full text, stores everything in a local SQLite database, and exposes 6 MCP tools that an AI agent can use to answer audit questions like:

- *"Which invoices are missing a matching purchase order?"*
- *"What documents support order 10687?"*
- *"Does this shipment match its related invoice?"*
- *"What are the contract terms for supply of goods?"*
- *"What inventory reports are available and what period do they cover?"*

---

## Stack

| Tool | Purpose |
|---|---|
| `pdfplumber` | Text extraction from text-based PDFs |
| `pytesseract` + `pdf2image` | OCR for image-based invoices (JPGs) |
| `SQLite` (built-in) | Document storage and metadata |
| `SQLite FTS5` | Full-text search across all documents |
| `FastMCP` | MCP server framework |
| `Python 3.12` | Runtime |

No cloud services. No vector database. No Docker. Runs entirely on localhost.

## Setup

### 1. Prerequisites

- Python 3.12+
- Tesseract OCR — download from https://github.com/UB-Mannheim/tesseract/wiki
  - Windows: install to `C:\Program Files\Tesseract-OCR` and add to PATH

### 2. Install dependencies

```bash
python -m venv venv

# Windows
.\venv\Scripts\Activate.ps1

# macOS / Linux
source venv/bin/activate

pip install -r requirements.txt
```

### 3. Run the data pipeline

Point `--data` at the root folder containing the five document category subfolders:

```bash
python pipeline.py --data /path/to/data
```

Expected output:
```
── invoices ──
  ✓ invoices/invoice_10248 | order=10248 | method=pdfplumber
  ✓ invoices/batch1-1486   | order=232855 | method=tesseract_image
  ...

Materializing cross-document relationships...
  Invoices missing a PO:   ['invoice_10436.pdf', 'invoice_10687.pdf', ...]
  POs missing an invoice:  ['purchase_orders_10372.pdf', ...]

✅ Pipeline complete — 45 documents ingested into db/knowledge_base.db
```

### 4. Run the MCP server

```bash
python mcp_server.py
```

The server starts on stdio transport and is ready for MCP client connections.

## MCP Tools

| Tool | Description |
|---|---|
| `search_documents` | Full-text search across all documents, with optional category filter |
| `get_document` | Retrieve full text and metadata for a single document by ID |
| `match_order` | Find all documents linked to a specific order ID across all categories |
| `find_mismatches` | Find documents missing an expected counterpart (e.g. invoice without a PO) |
| `list_inventory_periods` | List all inventory reports and the date periods they cover |
| `get_contract_terms` | Search for specific terms or clauses within contract documents |

Each tool returns structured results with `source` (filename) and `category` fields for full traceability back to the original document.

## Architecture overview

```
Raw files (PDFs + JPGs)
       │
       ▼
Format detection
  ├── Text PDF → pdfplumber
  └── Image / JPG → pdf2image + pytesseract (OCR)
       │
       ▼
Field extraction (regex)
  order_id · vendor · date · amount · period
       │
       ▼
SQLite storage
  ├── documents table (metadata + raw_text)
  ├── documents_fts (FTS5 full-text index)
  └── document_links (pre-computed order_id joins)
       │
       ▼
Relationship materialization
  has_matching_po · has_matching_invoice · has_matching_shipment
       │
       ▼
MCP Server (FastMCP, stdio)
  └── 6 tools → AI agent queries
```

## Data modeling decisions

**Order ID as join key.** Invoices, purchase orders, and shipping orders all share numeric order IDs in their filenames and document bodies. Cross-document matching is a simple equi-join on `order_id` — no fuzzy matching needed.

**Pre-computed mismatch flags.** The pipeline materializes `has_matching_po`, `has_matching_invoice`, and `has_matching_shipment` flags at ingestion time. The MCP tools read these flags directly rather than computing set logic at query time — keeping tool response times fast and the server stateless.

**SQLite FTS5 over vector search.** The dataset is 45 documents. SQLite's built-in full-text search handles all keyword and phrase queries at this scale. A vector database would add a 500MB+ model download, non-deterministic results, and significant setup complexity — unjustifiable for this dataset size.

**Plain text lineage.** Every document's extracted text is saved to `db/extracted/` as a `.txt` file. This makes it easy to inspect what the pipeline extracted and validate OCR quality without querying the database.

---

## Retrieval strategy

**Structured queries** (order ID lookups, mismatch detection) → SQLite direct queries on indexed fields. Used by `match_order` and `find_mismatches`.

**Full-text search** (vendor names, product terms, contract clauses) → SQLite FTS5 virtual table. Used by `search_documents` and `get_contract_terms`.

**Passage extraction** (contract terms) → After FTS match, the server finds the nearest passage around the query term and returns a 400-character window with context. This avoids returning raw multi-page contract dumps.

**Tradeoff acknowledged:** FTS5 is keyword-based — it won't match synonyms or paraphrases. For a production system, a hybrid approach (keyword + semantic vector search) would improve recall on the contract and inventory documents. For 45 documents and a 3–5 hour exercise, FTS5 is the right call.

---

## Known limitations

- **5 batch JPG invoices** (batch1-1486, batch1-1488, batch2-0998/0999/1000) have noisy OCR output. Order IDs could not be reliably extracted from 3 of them due to handwritten or degraded text. They are stored with `order_id = NULL` and flagged as unmatched.
- **Contract order ID** was incorrectly parsed from the filename date (20231231 → 20231). Contracts are not order-linked by design and are queried via `get_contract_terms` instead.
- **Vendor extraction** is regex-based and may miss vendors with non-standard formatting. A production pipeline would use a dedicated NER model.
